Architecture

OTLCBot is structured as four Python modules and one HTML frontend, backed by a single SQLite database. There is no framework beyond Flask for the webapp — the bot, agent, and weekly pipeline are plain Python scripts managed by systemd and cron.

Data Flow

The system has three main data flows, each operating on a different timescale:

Real-time: Message Logging & Agent Evaluation

Every message in a monitored chat hits bot.py's message handler, which writes it to the messages table and increments a per-chat counter. When the counter reaches the configured threshold (default: 30), the bot calls agent.py's run_agent_loop(). The agent gathers context from the database, asks Gemini to choose an action, executes the chosen tool, and logs the result to agent_actions. If the tool produces an observation, it's also saved to case_notes.

Weekly: Friday Pipeline

The cron-triggered weekly.py runs a sequential pipeline for each chat: build report → analyze sincerity → update group theme → generate image → send report → update user profiles → send DMs. For Owl Town, it then aggregates across all sub-chats, generates the prose gazette, and sends the combined report. Case notes from the past two weeks are injected into both profile and theme update prompts, closing the loop between short-term and long-term memory.

On-demand: Dashboard

The Flask webapp serves read-only API endpoints that query the database directly. The Telegram MiniApp fetches from these endpoints via the Cloudflare tunnel. Image proxying keeps the bot token server-side — the frontend requests /img/<id> and the backend resolves the Telegram file ID and streams the image.

Gemini API Usage

The system uses two Gemini models. gemini-2.5-flash-lite handles all text generation: recaps, profiles, case files, commentary, sincerity analysis, and the gazette. gemini-2.5-flash-image handles cartoon generation. Each call specifies a temperature(w) appropriate to the task — low for structured/analytical output, high for creative prose.

All Gemini calls receive the BOT_PERSONA system prompt, ensuring consistent voice. Many calls also receive a grounding block containing current database state (open bets, active profiles, group theme), which reduces hallucination(w) by anchoring the model's output to verified facts.
