Agent System

The agent system is OTLCBot's autonomous decision layer. Unlike the bot's command handlers (which respond to explicit user input) or the weekly pipeline (which runs on a fixed schedule), the agent evaluates the chat's current state and decides for itself whether to act. This is the core of what makes OTLCBot an agentic AI(w) application — the LLM is not just generating text on request but making autonomous decisions about when and how to intervene.

Two-Agent Architecture

The agent system is split into two independent processes: an Observer and a Speaker. They run on separate cron schedules and have no direct dependency on each other — the Observer writes to the database, the Speaker reads from it.

This separation means the bot accumulates observations continuously without over-posting. The Speaker reads the Observer's accumulated case notes as grounding context before each decision, giving it a richer picture of what has been happening between its own public actions.

Observer: Observe–Record Loop

For each configured chat, the Observer:

  1. Reads messages from the last 4 hours
  2. Skips the chat if fewer than 3 messages arrived (nothing worth noting)
  3. Asks Gemini to generate 1–3 internal case notes tagged as observation or discovery
  4. Writes the notes to the case_notes table (never visible to the group)
  5. Refreshes the group theme if 20+ messages arrived in the window

After all chats are processed, if any notes were filed, the Observer rebuilds the wiki (without Gemini) so People, Timeline, and Sincerity pages reflect the new data immediately.

Speaker: Observe–Reason–Act Loop

The Speaker runs every 3 hours via cron and also fires after AGENT_MSG_THRESHOLD messages (default: 50) accumulate in any chat via bot.py. For each chat:

1. Observe: Context Gathering

The gather_context() function assembles a snapshot from the database:

2. Reason: LLM Decision

The reason() function sends the context to Gemini with a dynamically-generated system prompt and asks it to choose an action. The model responds with structured output(w) — a JSON object specifying the action name, a one-sentence reason, and any parameters:

{"action": "send_commentary", "reason": "Chat active discussing movies, 6 messages since last action", "params": {}}

The system prompt is generated at runtime by build_system_prompt(), which iterates over the tool registry and assembles actions, descriptions, costs, and guidelines into the prompt. Adding a new tool automatically makes it available to the agent's reasoning.

The reasoning call uses low temperature (0.3) for consistent, conservative decisions. The default action is "nothing" — the agent is biased toward silence.

3. Act: Tool Execution

If the chosen action is not "nothing", the corresponding tool is looked up in the registry and executed. After a successful execution, the action is logged to agent_actions and added to the fired_this_cycle set so it cannot fire again in the same run. Quiet "nothing" decisions are intentionally not logged — the cooldown clock only resets on public actions.

Tool Registry

Tools are registered using the @register_tool decorator, which stores metadata alongside the function in a global TOOLS dictionary:

@register_tool(
    name="send_commentary",
    description="Send a brief observation about recent conversation.",
    guidelines="Only if chat active & bot quiet for a while.",
)
async def tool_send_commentary(conn, chat_id, bot, params):
    ...

Each tool declares guidelines (natural-language instructions included in the system prompt) and an optional cost hint. This is a form of tool use(w) where the model selects from a menu of functions based on their descriptions.

Tools in Detail

send_commentary

Reads the last 20 messages, group theme, and grounding block, then asks Gemini to write a single field observation (1–2 sentences). Uses high temperature (1.5) for expressive output. Saves the commentary to case_notes as type commentary.

illustrated_summary

Pulls conversation windows from the past 7 days (across all Owl Town chats if applicable), generates an AI cartoon via Gemini's image model, then posts it. Saves the image prompt to case_notes. The most expensive tool at ~$0.04 per invocation. Guidelines restrict it to once per week.

sincerity_check

Runs the DFW Sincerity Index analysis mid-week, triggered organically by the agent rather than on a fixed schedule. Posts the group-level result without individual scores.

add_media

Asks Gemini to scan recent messages for genuine media recommendations (not passing mentions). If found, checks for duplicates in the watchlist and adds the item. Uses low temperature (0.4) to avoid false positives. Uses JSONDecoder.raw_decode() to safely handle any trailing text in the model's response.

update_casefile

Asks Gemini to identify the single most personality-revealing moment in recent messages. If found, updates the target user's rolling profile, regenerates their case file dossier, saves a discovery note to case_notes, and announces the update to the group. The admin user is excluded from candidates.
