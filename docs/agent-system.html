Agent System

The agent system is OTLCBot's autonomous decision layer. Unlike the bot's command handlers (which respond to explicit user input) or the weekly pipeline (which runs on a fixed schedule), the agent evaluates the chat's current state and decides for itself whether to act. This is the core of what makes OTLCBot an agentic AI(w) application — the LLM is not just generating text on request but making autonomous decisions about when and how to intervene.

Observe–Reason–Act Loop

The agent follows a three-phase cycle, triggered after every AGENT_MSG_THRESHOLD messages (default: 30) in a monitored chat:

1. Observe: Context Gathering

The build_context() function assembles a snapshot of the chat's current state from the database:

Context fieldSourcePurpose
Message counts (6h / 24h / 7d)messages tableGauge activity level
Active users (last 6h)messages tableKnow who's talking
Recent messages (last 15)messages tableSee what's being discussed
Open betsbets tableIdentify stale bets to nudge
Hours since last actionagent_actions tableEnforce cooldown
Group themegroup_themes tableUnderstand the chat's personality
Recent observationscase_notes table (last 5)Remember what the bot has already said

This context is serialized as JSON and passed to Gemini as part of the reasoning prompt. The inclusion of prior case notes gives the agent continuity — without it, each evaluation would be amnesiac, potentially repeating observations or contradicting earlier commentary.

2. Reason: LLM Decision

The reason() function sends the context to Gemini with a dynamically-generated system prompt and asks it to choose an action. The model responds with structured output(w) — a JSON object specifying the action name, a one-sentence reason, and any parameters:

{"action": "send_commentary", "reason": "Chat has been active discussing movies for 2 hours", "params": {}}

The system prompt is generated at runtime by build_system_prompt(), which iterates over the tool registry and assembles the available actions, their descriptions, costs, and guidelines into the prompt. This means adding a new tool automatically makes it available to the agent's reasoning without any changes to the decision logic.

The reasoning call uses low temperature (0.3) to encourage consistent, conservative decisions. The default action is "nothing" — the agent is biased toward silence.

3. Act: Tool Execution

If the chosen action is not "nothing", the corresponding tool function is looked up in the registry and executed. Each tool receives the database connection, chat ID, bot instance, and any parameters from the reasoning step. After execution, the result (success or failure) is logged to agent_actions.

Tool Registry

Tools are registered using the @register_tool decorator, which stores metadata alongside the function in a global TOOLS dictionary:

@register_tool(
    name="send_commentary",
    description="Send a brief observation about recent conversation.",
    guidelines="Only if chat active & bot quiet for a while.",
    cost=0.0015,
)
async def tool_send_commentary(conn, chat_id, bot, params):
    ...

Each tool declares a cost (estimated Gemini API cost per invocation) and guidelines (natural-language instructions included in the agent's system prompt that help it decide when to use the tool). This is a form of tool use(w) where the model selects from a menu of available functions based on their descriptions.

Tools in Detail

send_commentary

Reads the last 20 messages, the group theme, and the grounding block, then asks Gemini to write a single field observation (1–2 sentences). Uses high temperature (1.5) for varied, expressive output. Saves the commentary to case_notes as type commentary.

illustrated_summary

Pulls conversation windows from the past 7 days (across all Owl Town chats if applicable), generates an AI cartoon via Gemini's multimodal image model, then generates a caption. Sends the image and caption together. Saves the caption to case_notes as type observation. The most expensive tool at ~$0.04 per invocation.

nudge_bet

Finds open bets older than 14 days and sends a reminder to the group. A simple, deterministic tool that doesn't require Gemini — it just queries the database and formats a message.

sincerity_check

Runs the same DFW Sincerity Index analysis used in the weekly reports but triggered organically by the agent rather than on a schedule. Posts the group-level result without individual scores.

add_media

Asks Gemini to scan recent messages for genuine media recommendations (not passing mentions). If found, checks for duplicates in the watchlist and adds the item. Uses low temperature (0.4) to avoid false positives.

update_casefile

Asks Gemini to identify the single most personality-revealing moment in recent messages. If found, updates the target user's rolling profile, regenerates their case file dossier, saves a discovery note to case_notes, and announces the update to the group. The admin user is filtered out of candidates.
