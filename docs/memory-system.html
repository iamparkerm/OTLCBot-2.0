Memory System

Large language models are stateless — each API call starts with no memory of previous calls. They also have a limited context window(w), meaning they can only process a fixed amount of text per request. OTLCBot addresses both constraints with a three-tier memory architecture that stores observations at different granularities and consolidates them over time.

Tier 1: Case Notes

Case notes are the bot's short-term memory — raw observations recorded as the agent acts during the week. Three agent tools write case notes:

ToolNote typeTarget userContent
send_commentarycommentaryNone (group-level)The commentary text posted to chat
update_casefilediscoveryThe profiled userWhat was revealed about the user's personality
illustrated_summaryobservationNone (group-level)The caption generated for the cartoon

Case notes serve three downstream purposes: they give the agent continuity between firings (it reads its last 5 notes before each decision), they feed into the weekly gazette prompt, and they flow upward into Tier 2 during Friday consolidation.

Tier 2: Profiles and Themes

Tier 2 is the bot's long-term memory — consolidated summaries that compress weeks of observation into a few hundred words. This is necessary because the raw data (thousands of messages, dozens of case notes) would far exceed the LLM's context window. The consolidation process is a form of abstractive summarization where the model produces a new, shorter text that captures the essential patterns rather than excerpting verbatim.

User Profiles

Each user has a rolling profile_text (a personality summary) and a case_file_text (a formatted dossier for the dashboard). Both are regenerated weekly by update_user_profile() and generate_case_file_text().

The profile update prompt receives three inputs:

  • The existing profile (if any) — the model's prior understanding
  • This week's message snippets — what the user actually said
  • Recent discovery notes from case_notes (last 14 days) — what the agent previously noticed about this user

The prompt instructs: "Consolidate and merge — don't just append. Drop stale details that haven't recurred." This prevents the profile from growing indefinitely while ensuring recurring patterns are reinforced and one-off observations decay naturally.

Profiles are versioned (incremented each week). At milestone versions (2, 4, 8, 13, 26), the bot generates and posts an announcement to the group.

Group Themes

Each chat has a rolling theme_text that captures its culture: running jokes, recurring references, group dynamics, shared interests, and communication style. The update process mirrors user profiles — the existing theme, this week's conversation windows, and recent case notes are all injected into the consolidation prompt.

Group themes are used throughout the system: they appear in the agent's context when reasoning, they're injected into image generation prompts for visual consistency, and they're part of the grounding block for creative text generation.

Tier 3: Execution Log

The agent_actions table records every agent decision: the chosen action, the reasoning, the timestamp, and whether execution succeeded. Its primary purpose is cooldown enforcement — the agent checks "hours since last action" before deciding. It also serves as a debugging and audit trail.

The Upward Flow

The critical design pattern is the upward flow from Tier 1 to Tier 2. Without it, the tiers would be independent silos — the agent would make observations during the week that the weekly consolidation would never see.

The flow works as follows:

WhenWhat happens
Agent fires mid-weekObservation saved to case_notes (Tier 1)
Friday: update_user_profile()Pulls discovery notes about this user from last 14 days and injects them alongside message snippets
Friday: update_group_theme()Pulls all case notes for this chat from last 14 days and injects them alongside conversation windows
ResultTier 2 profiles/themes now reflect both raw messages and the agent's curated observations

This means the bot's long-term memory is richer than what raw messages alone would produce. If the agent noticed that "Subject keeps bringing up sourdough bread" during the week, that observation will be absorbed into the user's profile on Friday — even if the individual messages mentioning bread were too scattered to form a clear pattern on their own.

The Grounding Block

When the system needs to give Gemini awareness of current state (for commentary, recaps, or image generation), it assembles a grounding block via build_grounding_block(). This is a form of retrieval-augmented generation(w) that reduces hallucination(w) by injecting verified facts into the prompt.

The grounding block includes:

  • The group theme (from group_themes)
  • Up to 5 open bets (from bets)
  • Recent watchlist items from the last 14 days (from watchlist)
  • Active user profile headlines (from user_profiles)

This ensures the model can reference real, current information — actual bet descriptions, actual watchlist items, actual personality observations — rather than inventing plausible-sounding details.
